## 서론

### 연구 배경 및 필요성

현대의 소프트웨어 공격은 실행 흐름 변조를 기본으로 한다.
실행 흐름 변조는 메모리 변조로부터 시작된다. 

ARM에서 하드웨어를 사용한 [메모리 보호기술](https://www.qualcomm.com/content/dam/qcomm-martech/dm-assets/documents/pointer-auth-v7.pdf)을 선보였으나, 개발 초기단계이므로 해당 기술이 적용된 사례는 많지 않다.

하드웨어 보안 기술이 적용된 기기는 늘어날 예정이다. 
아직은 실제 사용 환경에서의 보안에 관한 연구가 많이 이루어지지 않았다. 실제 적용된 하드웨어에 관한 보안 연구도 다양하게 이루어지지 않았다.

### 선행 연구의 한계점

메모리 보호를 소프트웨어적으로 해결하는 연구가 상당수 진행되고 있으나 성능의 저하가 극심함으로 인해
실제 프로그램에 적용하기에는 한계가 존재한다.

### 연구의 목적

ARM 하드웨어 보안 기술이 적용되어 있는 하드웨어의 적용 방식과 보안성에 대한 진단을 수행한다.

## 관련 연구

### 메모리 보호 기술

- AddressSanitizer: 할당된 메모리를 벗어나는 것을 확인하여 다른 값으로 덮어씌워지는 것을 방지
- 메모리 태깅: 메모리에 태그를 주어 특정 메모리 영역이 아닌 부분을 사용하는 것을 방지

- CCFI: 메모리 내부 포인터의 변조를 해시 등으로 보호
- ZeR∅: 캐시 내부에 메타데이터를 넣어 메모리의 변조를 방지
- Morpheus: RISC-V를 사용하여 메모리 도메인을 만들고 그에 대한 태깅, 변위 및 암호화로 포인터 변조 방지

하드웨어를 사용하여 메모리 보호 기술을 기존의 소프트웨어 방식보다 빠르게 하는 방향으로 변화 중

- PAC: 포인터 인증 방식, 포인터를 사용하기 전에 인증하여 포인터의 무결성을 보장한다.

### Pointer Authentication

PA. 포인터에 암호학 서명을 넣어 무결성을 보장한다.
상위 특정 비트에 PAC (Pointer Authentication Code)를 넣고 사용할 때 검증하여 변조 여부를 확인 할 수 있다.

ARM v8.3-A에서 새로 추가됨.

PAC는 128bit key, 64bit pointer, 64bit 수정자를 통해 포인터에 서명한다.
표준에서는 QARMA라는 블록 암호를 이용하여 PAC를 계산할 것을 권장한다.

포인터 종류에 따라 key가 다르며 5개까지 있다. 수정자는 일반적으로 스택 주소를 사용한다.

PAC key
- code pointer: APIA, APIB
- data pointer: APDA, APDB
- general purpose: APGA

여러 개의 키를 사용하는 이유는 하나의 인증된 포인터가 다른 포인터로 대체되는 포인터 대체 공격을 보호하기 위함이다.
키는 시스템 레지스터에 저장되어 User mode인 EL0에서는 읽을 수 없다. => 유저는 PAC 서명에 사용되는 키를 읽거나 변경 불가능. [[ref]](https://googleprojectzero.blogspot.com/2019/02/examining-pointer-authentication-on.html)


PAC 사용을 위해 ARMv8.3-A부터는 PAC으로 시작되는 명령어와 AUT로 시작되는 명령어가 추가되었다.

PAC로 시작하는 명령어는 포인터에 PAC를 부여한다.
AUT로 시작하는 명령어는 PAC가 추가된 포인터를 검증하여 포인터를 사용할 수 있도록 한다.

리턴이나 점프 등 상황에서 사용할 수 있는 RETAA 명령어도 추가되었다.

![image](https://github.com/neutrinox4b1/neutrinox4b1.github.io/assets/101745344/cceb8ddf-59c3-4ff3-a570-25b34fadfcd7)

macOS에서 PAC는 포인터에 주소가 대입되는 시점에 계산하는 Forward edge,
함수 종료 후 리턴 주소를 보호하는 Backward edge 두 가지의 보호를 지원한다.

### PAC 기반 메모리 보호기술

#### PAC를 활용한 방어연구

P. Nasahl 외 2명은 PAC를 이용하여 fault attack을 통한 간접 점프 공격을 보호하는 방법을 제시하였다. 점프하려고 하는 주소와 간접 주소 분기점 두 가지를 보호하여 공격자가 해당 주소를 실행하는 것을 방지한다.

PAL은 커널 내부 실행 흐름 보호 방식이다. PAC를 사용하여 보호한다.

A. Fanti 외 4명은 LLVM을 활용해 메모리에 레지스터가 저장될 때 PAC를 사용하여 레지스터의 값을 보호하는 방식을 제안한다.

PTAuth는 힙에서 temporal memory corruption을 방지하기 위해 PAC를 사용하여 포인터의 무결성을 보장한다. 컴파일러를 수정하였다.

PACMem은 PAC 명령어를 활용하여 객체 서명과 크기를 담은 메타 데이터를 생성, 이를 활용하여 객체의 temporal safety와 spatial safety를 보장한다.

#### PAC를 공격하는 연구

PACMAN은 커널에서 PAC를 인증하고 실행하는 특정한 gadget을 이용하여 PAC에 대한 검증을 오류 없이 진행, 무차별 대입으로 실제 수행되는 PAC의 값을 알아낸다. 맞는 지 검증하기 위해서는 부채널 공격을 수행하였다.
2.94분 이내로 하나의 포인터에 대한 PAC를 알아낼 수 있고 정확도는 90%이다.

에뮬레이션 된 PAC를 분석한 PAC it up에서는 포인터 생성 이전의 포인터 값과 수정자를 이용하여 원하는 PAC서명을 얻을 수 있다고 하였다.
이를 통해, 서명된 포인터를 저장했다가 필요할 때 같은 수정자를 사용하는 함수에서 해당 포인터를 사용하여 서명 인증을 회피할 수 있다.


## PAC 생성 및 관리 과정 분석

M1에서 PAC이 어떻게 생성되고 관리되는지를 서술함.


XNU 커널에서의 PAC는 부팅 시에 생성된다. CPU 권한 레벨과 PAC키의 레지스터를 이용하여 PAC키를 생성한다. 생성 이후에는 시스템 레지스터에 저장된다.

유저가 사용하는 PAC키는 프로세스별로 분리된다. -> 같은 프로세스의 스레드는 같은 키를 공유한다.
APIA, APDA키는 주소 공간 공유영역 할당시 생성, APIB, APDB는 프로세스 생성 시 생성된다.

![image](https://github.com/neutrinox4b1/neutrinox4b1.github.io/assets/101745344/f8b67037-cdcc-4279-99df-0480844a5fa2)

유저가 사용하는 PAC키는 task 구조체에서 관리한다.
task 구조체 내부 rop_pid, jop_pid 변수는 키를 저장한다.

APIA, APDA -> jop_pid
APIB, APDB -> rop_pid

### PAC key 관리

커널의 PAC 키는 기본적으로 시스템 레지스터에 저장되나, 코어 절전이나 인터럽트같이 코어 내부 상태를 저장하는 경우 커널메모리에 커널 PAC키가 저장된다.

$\therefore$ 커널 PAC키는 부팅 후 변경되지 않으므로 임의의 커널메모리를 읽을 수 있으면 PAC 키 값도 읽을 수 있다.

![image](https://github.com/neutrinox4b1/neutrinox4b1.github.io/assets/101745344/96120731-9964-4695-8d1a-92074f367a77)

유저 프로세스에서..

A키는 공유영역 id가 같은 경우 같은 키를 사용하고
B키는 fork 같이 메모리 주소 공간을 공유하는 경우 같은 키를 사용한다.

### PAC 사용 과정

macOS에서 arm64e_preview_abi를 활성화하고 arm64e 아키텍처로 컴파일 설정해야 한다.

#### Forward Edge

```C
void foo();
void bar();

int main()
{
    void (*ptr1)(void) = foo;
    void (*ptr2)(void) = bar;
    ptr1();
    ...
}
```

forward edge가 보호되는 상황.
함수 포인터가 대임되는 시점에서 PAC이 계산된다. 계산에 사용되는 인자는 함수 포인터(ptr), IA 키, 수정자 0이다.

이후에 함수 포인터를 사용하면 포인터 무결성을 체크하고 함수를 실행.

#### Backward Edge

ARM64에서 리턴주소는 X30(LR, Link Register)에 저장된다. 
함수 내부에서 또 다른 함수가 호출되면 그 리턴주소는 스택에 저장한다.

```C
void foo() 
{
    ...
}

void bar()
{
    foo();
}
```

bar 함수의 프롤로그에서 IA key, sp, ret주소를 이용하여 생성된 주소(PAC)를 스택에 저장한다. (sfp 뒤에 저장됨)

foo로 분기 후 복귀까지 끝나면 bar의 에필로그에서 저장된 주소를 x30레지스터에 불러오고 retaa 명령어로 return과 동시에 포인터 검증을 수행한다. canary와 비슷한 느낌.

### PAC 오류 처리

PAC 인증 실패 시?
인증 성공한 포인터에서 오류 비트가 추가된다.

오류가 발생하는 부분은 PAC를 인증하는 곳이 아니라 인증 실패된 포인터를 사용하는 부분에서 일어난다.

오류 비트는 상위 3bit에 1로 표시된다.
발생하는 오류는 페이지 오류로 나타난다. => 인증 실패 포인터에서 오류 비트 변경 -> PAC 처리된 포인터를 제약없이 사용 가능하다.

## PAC에 대한 공격

### 무차별 대입 공격
PAC 크기는 7bit, 추측하는데에는 2^7이다.
그러나 PAC을 생성할 때 포인터 주소, 스택주소, 키 3가지 요소가 존재하며 매번 PAC가 생성되므로 모든 PAC의 경우를 하나씩 대입하는 것으로는 공격 성공이 어렵다.

유저 프로세스는 무한정 fork가 가능하므로 같은 키 프로세스를 무한정 생성 가능하여 성공 가능성이 존재.

### C++ 가상함수 공격

가상함수 테이블은 APDA, 가상함수 포인터는 APIA 키로 보호되어있다.

각자의 수정자는 클래스와 함수의 고유한 해시값으로 고정되어있다.
공격자가 PAC 서명 명령어를 사용 가능하면 원하는 함수를 PAC 서명을 통해 실행할 수 있다.

1. 공격자가 원하는 가상함수 테이블을 생성
2. 목표로 하는 객체 가상함수의 해시값을 알아냄 -> 객체, 가상함수의 이름에서 얻을 수 있고 컴파일 시기에 고정된값.
3. 해시값을 수정자로 하여 공격자가 만든 함수포인터, 테이블을 PAC 명령어로 서명한다.
4. 악의적인 포인터는 정상적인 인증과정을 거쳐 사용 가능하다.
5. 서명한 가상함수 테이블의 주소 포인터로 덮어씌우면 가상함수를 불러서 원하는 함수를 수행가능
